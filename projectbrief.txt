1 The “Why” — The Real Problem
The Status Quo
The Status Quo
In most academic institutions, students struggle to know when and how they can meet a professor. Communication relies on fragmented methods: static office-hour PDFs, email back-andforth, hallway guesswork, or word of mouth.
Core Pain Points
• Availability Uncertainty: Students don’t know if a professor is actually available, busy with
another student, working remotely, or gone for the day.
• Wasted Time & Friction: Students walk to offices only to find doors closed, or wait without
knowing how long the queue is.
• Inequitable Access: A small group of “power users” often dominates access, while others
miss opportunities due to lack of visibility.
• Manual Coordination Overhead: Professors waste time answering repetitive “Are you free?”
emails.
The Goal
Design and build a system that provides clear, real-time visibility into professor availability,
supports fair access, and reduces coordination friction.
This is a product problem first — not just a coding exercise.

2.1 Functional Requirements (MVP)
2.1.1 User Roles & Interfaces
Professors (Admin View)
• Must have a secured management interface (authentication required).
1
• Can:
– Update availability status quickly
– View and manage their current queue
– Control their schedule

Students (User View)

– Estimated wait times
• Must have a booking flow that handles:
– Requests

2.1.2 Interaction & Queuing
• Students can request a slot or join a virtual queue when the professor is unavailable.
• The system must handle cancellations automatically and fairly, without manual intervention.
A booking is not a single action — it is a stateful process.
2.1.3 Fairness Rules (Business Logic)
The system must enforce at least one fairness constraint on the server side, such as:
• Maximum concurrent bookings per student
• FIFO queue ordering
• Cool-down periods between meetings
Bonus (Optional):
Implementing additional fairness rules beyond the first — and clearly documenting their purpose and tradeoffs — will be considered a bonus.
2
Critical Constraint: The “Server Authority” Rule
All fairness rules and queue logic must be enforced by a trusted server (e.g., Backend
API, Database Trigger, or Serverless Function).
Why this matters: Any logic that runs only in the frontend can be bypassed.
Requirement: If a student bypasses the UI and sends a forged request (e.g., via curl or
Postman), the backend must reject it.
UI validation improves UX.
Server-side validation determines your grade.
2.2 Non-Functional Requirements
• Real-Time Accuracy:
Status updates must propagate automatically without page refresh. Acceptable approaches
include polling, WebSockets, or server-sent events. Latency between a server state change
and the client UI update must be under 5 seconds for all connected clients simultaneously
(e.g., if Student A books a slot, Student B’s screen must update to ’Booked’ within 5s)..
• Data Integrity:
Information shown to users must be consistent and race-condition safe.
• Deployment:
The application must be deployed to the cloud and accessible via a public URL (e.g., https:
//project.vercel.app). Deployment should use free-tier friendly services (e.g., Vercel, Netlify,
Supabase, Firebase, Railway). Ensure your chosen platform does not ’sleep’ aggressively
during your demo slot.
• Access Control:
While the login page is public, all internal pages (dashboards, bookings, admin views) must
be protected by authentication and role-based access control.
Explicitly Out of Scope
To ensure focus on high-value system logic, the following are not required:
1. External calendar integrations (Google / Outlook)
2. Native mobile apps (React Native / Swift / Kotlin)
3. In-app chat or messaging
4. A public ’Sign Up’ page is not required; a simple Login page is sufficient. The system
should assume users are pre-loaded via the Ops Lead’s seeding script.
These are intentionally excluded to keep the focus on coordination, fairness, and realtime systems, not integrations.

A Appendix
A.1 Interaction & Queuing (State Transitions)
A booking flows through states. It is never just “created” or “deleted.”
A.1.1 1. Request or Queue
Open Scenario
• Professor status: Available
• Student sees “Book Now”
• Action → Slot locked → Status becomes Booked
Busy Scenario
• Professor status: Busy or all slots filled
• Student sees “Join Queue”
• Action → Student added to queue with a position
A “slot” may represent a fixed time window, a rolling availability block, or a single active
meeting.
The modeling choice is yours — but it must be consistent and defensible.
A.1.2 2. Handling Cancellations (Required Logic)
Avoid:
Making the slot immediately public, causing race conditions.
Required Behavior:
1. Booking is cancelled
2. System checks queue
3. If queue is empty → slot becomes Open
4. If queue exists → next student is automatically promoted
5. The promoted student is notified
8
Notifications may be implemented via email, UI alerts, or logged system messages.
The grading focus is the backend decision, not the delivery channel.
For the purpose of grading, immediate UI feedback (e.g., a toast notification or dashboard
update) is preferred over email integration to ensure testability during the demo. Email
integration is optional and should not delay the core deliverable.
You must decide how to handle a promoted student who does not ’claim’ their slot within X
minutes (stale state). Does the system skip them? Does the slot remain blocked?" (This forces
them to think about timeouts).
A.2 Fairness Rules (Server Authority Test)
You must implement at least one fairness rule.
For each rule you choose:
• Enforce it on the server
• Document at least one alternative you considered and rejected, with reasoning
Examples:
• Max concurrent bookings
• FIFO queue ordering (with timeout logic)
• Cool-down periods between meetings

Queues & Booking
Q: When a student joins a queue, is it instant (one click), or do they need to provide details?

A: Input is required. The student should fill out a short form (e.g., "Reason for meeting" or "Topic") before entering the queue.

Why: The professor needs context (e.g., "Grade Dispute" vs. "Project Help") to prioritize or prepare for the meeting.

Q: Is the system only for "Live" queuing, or can students book future dates?

A: The system must support both modes:

Future Scheduling: A student can book a specific 15-minute slot for next Tuesday while browsing on the weekend.

Live Queue (Drop-ins): A student can join a "First-Come-First-Served" line during open office hours.

Key Takeaway: Your data model needs to handle Appointments (Time-bound, future) and Waitlist Entries (Order-bound, immediate).
