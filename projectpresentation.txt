CRITICAL: Server Authority
Trust No One.
(Especially the Client)
▶ Fairness rules must be enforced on the SERVER.
▶ Frontend checks are for UX only.
▶ Deliverable 3: We will try to bypass your API (Postman/Insomnia).
Non-Functional Constraints
▶ Real-Time: < 5 seconds latency for all clients (WebSockets/Polling/SSE). No
page refreshes.
▶ Data Integrity: Race-condition safe writes; UI must match server truth.
▶ Deployment: Public URL (Vercel, Railway, etc.). Cloud-native.
▶ Security: All internal pages protected. Role-based access.
Access Control and Deployment
▶ Public Login page allowed; everything after authentication must be protected.
▶ Role-based access: admins (professors) vs. students with least-privilege
defaults.
▶ Choose cloud hosting that stays awake during demos (Vercel, Railway,
Render, Supabase/Firebase).
▶ Seed users via ops script; no public sign-up required.
Interaction and Queueing
▶ Booking is a stateful process (request →confirm →booked/queued
→done/cancelled).
▶ If busy, students join a virtual queue with clear position and ETA.
▶ Cancellations auto-check the queue and promote the next student without
manual steps.
▶ Server rejects forged requests (curl/Postman) that bypass UI limits.
Fairness Rules (Server-Side)
▶ Enforce at least one server-side rule: max bookings, FIFO queue, or
cooldown.
▶ Document one alternative rule you considered and rejected (with tradeoffs).
▶ Bonus: layer multiple rules (e.g., FIFO + cooldown) and log decisions for
auditability.
▶ All checks must run server-side before writes; UI validation is supportive only.
Cancellation Flow (Required)
1. Booking cancelled.
2. Server inspects queue.
3. If queue empty: slot becomes Open.
4. If queue exists: auto-promote next student and notify.
5. Handle stale promotions (timeout/skip) to avoid blocked slots.
