A. Functional Requirements
1. Live Deployed “Walking Skeleton”

A minimal version of the system deployed to a public cloud environment (e.g., Vercel, Netlify, Heroku, AWS).

The application must be accessible via a public URL (not localhost).

2. Authentication (Login Only)

A functional Login Screen that accepts email/password.

Role Separation:

Student View: Logging in as a student shows a student-specific interface (e.g., "Book a Slot").

Professor View: Logging in as a professor shows a professor-specific interface (e.g., "Manage Availability").

Note: "Sign Up" is NOT required. You will log in using the pre-seeded accounts.

3. Database Connectivity & Seeding

The database must be connected and populated using a seed script.

Minimum Dataset:

5 Professors (with distinct names/departments).

20 Students (with valid credentials).

B. Submission Requirements

Student A: (e.g., student1@uni.edu / pass123)

3. The "Data Proof" Video (Critically Important)

A 3–5 minute screen recording that strictly follows this sequence to prove the database is live:

The Login: Log in as "Professor A." Show the dashboard.

The Mutation (The Proof):

Open your Database Dashboard (e.g. Supabase Table Editor, PGAdmin, or Console).

Find "Professor A" in the users or profiles table.

Manually change their name (e.g., change "Dr. Jones" to "Dr. SUPERMAN").

Save the change in the database.

The Verification:

Switch back to the live website.

Refresh the page.

Success: The UI now displays "Dr. SUPERMAN," proving the data is being fetched dynamically and is not hardcoded in the frontend.
